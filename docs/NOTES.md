# Notes

- **⚠️ IMPORTANT — lost-wakeup race in `WorkerRunner#claimLoop`.** In `packages/worker/src/internal/worker-runner.ts`, between `scheduler.claim()` returning `null` (queue empty) and `#waitForNotification()` installing `#wakeResolve`, there is an `await` gap. A `PUBLISH`-driven `wake()` that lands in that window finds `#wakeResolve === null` and is **silently dropped** — the just-enqueued job then waits until the next poll (`pollInterval`, default **5000ms**) instead of being picked up in ~1ms. Measured ~0.7–1.3% of tight back-to-back enqueues hit this (POC: `tmp/pickup-poc.mjs`, with `pollInterval=250ms` victims land at ~252ms). An idle-gap enqueue never hits it (the worker is reliably parked first); the window only opens when an enqueue races the worker's claim round-trip — i.e. high-churn queues. Fix options: re-check the waiting list once after parking, or have `wake()` set a "pending wake" flag that `#waitForNotification` consumes before sleeping. Mitigation until fixed: lower `pollInterval` to cap the worst-case stall. Correctness is unaffected (leases + stalled-recovery still deliver) — this is purely a latency tail.

- **Batch claiming — throughput direction & design open questions (post-v0.1).** Today every job is two sequential Redis round-trips on one command connection (claim + complete), and the claim loop awaits each claim before starting the next, so concurrent claims never pipeline. This serialization — not Lua or Redis — is most of the processing-throughput gap vs BullMQ (~2,500–4,400 no-op jobs/s vs ~15,000; see `docs/DECISIONS.md` table). Direction: claim in batches to cut round-trips. Captured questions/decisions so far:
  - **Batch-claim ≠ prefetch.** The SPEC's rejected thing (`docs/SPEC.md` "No Prefetching (v0.1 Constraint)") is _buffered prefetch_ — claiming more than free capacity and parking jobs locally, which breaks the `active == executing` invariant, raises false-stall risk, and complicates shutdown. Claiming **up to the number of free semaphore permits in one Lua call and dispatching all immediately** keeps the invariant (no active-but-idle jobs) while still collapsing N round-trips into 1. Pursue this; keep buffered prefetch rejected.
  - **Cheapest first step, no Lua change:** stop awaiting each claim before the next — issue up-to-free-permits claims concurrently so node-redis auto-pipelines them. Worth measuring on its own (POC: `tmp/pickup-poc.mjs`) before investing in batch Lua.
  - **Batch size is its own capped knob, NOT `= concurrency`.** A single Lua call for, e.g., 200 jobs is bad: Redis is single-threaded and runs Lua atomically, so a big claim (~5 ops/job → ~1000 ops) blocks _every_ client for ~1ms+ (past `lua-time-limit` it replies `-BUSY`); the reply ships N full job hashes incl. payloads (fat bulk reply); and a greedy worker starves the others (fairness). Round-trip savings also have steep diminishing returns — batch 1→10 removes ~90% of trips, 10→100 only ~9% more. So claim `min(freePermits, claimBatchCap)` with a modest cap (gut: ~16). Batching the **completes/acks** is likely a bigger win than pushing claim batch higher.
  - **Should the cap be user-exposed?** Lean yes, but _quietly_ (optional advanced setting, great default, one line of docs). Test = "does the optimal value vary across real deployments by orders of magnitude?" — and it does, along two axes the library can't know: **Redis RTT** (loopback 0.3ms vs serverless/cross-region 10–50ms → optimal batch much larger when RTT is high; the dominant axis) and **payload size** (1MB payloads want batch≈1 to keep replies small). Tune-up cases: high-RTT Redis, firehose of tiny fast jobs, one dominant worker. Tune-down cases: big payloads, long handlers (round-trips are noise), many workers few jobs (fairness), shared/multi-tenant Redis (keep scripts tiny).
  - **Safe to expose because claim-to-capacity is ceilinged by concurrency** — a worker can never claim more than free permits, so the knob is a pure round-trip optimization that can't create active-but-idle jobs. Still **clamp internally to a hard max** (~64–128) regardless of user value, to protect Redis from a footgun blocking script.
  - **Natural end state is adaptive:** grow the batch when claims keep returning full (deep queue, falling behind), shrink when partial — default just works across RTTs. More than v0.1 wants, but exposing a manual cap now doesn't block it: the manual value becomes the ceiling of the adaptive range. Next concrete step when picking this up: benchmark a batch-size sweep (1,2,4,8,16,32,64,128) measuring throughput _and_ p99 pickup + script blocking to find the knee empirically rather than guessing the default.

- Worker wakeups use regular `PUBLISH`/`SUBSCRIBE` on `{q:<queueId>}:notify` (fire-and-forget; reliability comes from leases + stalled-recovery + `pollInterval`, not the channel). In Redis Cluster, `PUBLISH` fans out to every shard via the cluster bus; when targeting clusters at scale, switch to sharded pub/sub (`SPUBLISH`/`SSUBSCRIBE`, Redis 7.0+) — the channel name hashes to a slot like a key, so the message stays on the shard that owns the queue data and its subscribers. Same semantics, no cluster-bus flood. Mostly a command-name swap plus making the client cluster-aware; works fine on standalone Redis too. Keep regular pub/sub for v0.1; revisit when clustering is a real target.
