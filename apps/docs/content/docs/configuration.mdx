The `@panqueue/config` package is the shared contract for a Panqueue app. Put
queue IDs, payload types, Redis connection options, and queue settings in one
file, then import that file from producers and workers.

  <Card

  />
  <Card

## Define a config

<Steps>
  <Step>
    Create a `panqueue.config.ts` file that exports one shared config.

    ```ts title="panqueue.config.ts"
    import { definePanqueueConfig } from "@panqueue/config";

    type Queues = {
      email: {
        to: string;
        subject: string;
        body: string;
      };
      thumbnail: {
        imageUrl: string;
        width: number;
      };
    };

    export const pq = definePanqueueConfig<Queues>({
      redis: { url: "redis://localhost:6379" },
      queues: {
        email: {},
        thumbnail: {},
      },
    });
    ```

  </Step>

  <Step>
    Import the config anywhere that needs queue types.

    ```ts title="enqueue.ts"
    import { QueueClient } from "@panqueue/client";
    import { pq } from "./panqueue.config.ts";

    await using client = new QueueClient(pq);

    await client.enqueue("email", {
      to: "user@example.com",
      subject: "Welcome",
      body: "Thanks for signing up.",
    });
    ```

    ```ts title="workers/email.ts"
    import { defineWorker } from "@panqueue/worker";
    import { pq } from "../panqueue.config.ts";

    export const emailWorker = defineWorker(pq, "email", async (job) => {
      await sendEmail(job.data);
    });
    ```

  </Step>
</Steps>

`definePanqueueConfig` is an identity function. It returns the object unchanged,
opens no Redis connections, and exists so TypeScript can infer queue IDs and
payload types from the same value you pass to `QueueClient`,
`defineWorker`, and `WorkerPool`.

## Queue payloads

The generic passed to `definePanqueueConfig` maps queue IDs to payload types.
Payloads must be JSON-serializable: strings, numbers, booleans, null, arrays,
and plain objects made from those values.

```ts
type Queues = {
  email: {
    to: string;
    subject: string;
    body: string;
    templateId?: string;
  };
  resizeImage: {
    sourceUrl: string;
    sizes: number[];
  };
  syncUser: {
    userId: string;
    reason: "created" | "updated" | "deleted";
  };
};
```

Avoid `Date`, `Map`, `Set`, functions, class instances, and binary values in
payloads. Serialize them into plain JSON values before enqueueing.

<Callout type="info" title="Types flow from the config">
  If you enqueue to `resizeImage`, TypeScript expects the `resizeImage` payload. If you define a
  worker for `email`, `job.data` is the `email` payload.
</Callout>

## Redis connection

`redis` accepts the same connection options everywhere Panqueue opens Redis.

  <Tab value="URL string">

    ```ts
      redis: "redis://localhost:6379",
      queues: {
        email: {},
      },
    });
    ```

  </Tab>
  <Tab value="URL object">

    ```ts
    export const pq = definePanqueueConfig<Queues>({
      redis: { url: "redis://localhost:6379" },
      queues: {
        email: {},
      },
    });
    ```

  </Tab>
  <Tab value="Host options">

    ```ts
    export const pq = definePanqueueConfig<Queues>({
      redis: {
        host: "localhost",
        port: 6379,
        db: 0,
        tls: false,
      },

        email: {},
      },
    ```


  </Tab>
</Tabs>

## Queue settings

Every key in `queues` must match a key in the `Queues` type. Use an empty object
for default behavior.

```ts
type Queues = {
  email: { to: string; subject: string };
  image: { url: string; width: number };
};

export const pq = definePanqueueConfig<Queues>({
  redis: "redis://localhost:6379",
  queues: {
    email: {
      retention: {
        failed: { ttl: 86_400_000 },
      },
    },
    image: {},
  },
});
```

| Option      | Type     | Default   | Description                                    |
| ----------- | -------- | --------- | ---------------------------------------------- |
| `retention` | `object` | See below | Controls how long finished jobs stay in Redis. |

## Retention

Every finished job is a Redis key. Without retention, completed and failed jobs
accumulate forever and eventually exhaust memory. Retention rules tell Panqueue
when to evict finished jobs from Redis, so storage stays bounded.

Each queue defines a policy per terminal state — `completed` (job succeeded)
and `failed` (job exhausted retries or stalls).

### RetentionRule

A `RetentionRule` controls what happens to a finished job. It can be:

| Value               | Behavior                                                                |
| ------------------- | ----------------------------------------------------------------------- |
| `false`             | Delete the job immediately on finish.                                   |
| `true`              | Keep the job forever (unbounded growth — use with caution).             |
| `{ ttl: number }`   | Keep for at most `ttl` milliseconds before eviction.                    |
| `{ count: number }` | Keep at most `count` finished jobs (oldest evicted first).              |
| `{ ttl, count }`    | Keep jobs that satisfy both bounds. Eviction runs atomically on finish. |

### Defaults

| State       | Default                             | Rationale                               |
| ----------- | ----------------------------------- | --------------------------------------- |
| `completed` | `false`                             | Successful jobs rarely need inspection. |
| `failed`    | `{ ttl: 604_800_000, count: 1000 }` | Bounded dead-letter for debugging.      |

Set custom retention per queue:

```ts
export const pq = definePanqueueConfig<Queues>({
  redis: "redis://localhost:6379",
  queues: {
    email: {
      retention: {
        completed: { count: 5000 }, // Keep last 5000 successes
        failed: { ttl: 86_400_000, count: 100 }, // Keep failures for 24h
      },
    },
  },
});
```

<Callout type="warn" title="Worker concurrency is not config-level">
  The concurrency option on `defineWorker` (e.g. `{ concurrency: 5 }`) controls how many
  jobs a single worker process runs in parallel. That is a process-level concern — not a
  queue-level setting inside `@panqueue/config`. Use the config for queue shape and
  queue-level settings; use worker options for handler concurrency, polling, leases,
  recovery, and event handlers.
</Callout>

## Share it across packages

A small Panqueue project can keep the config at the repository root:

```txt
.
├── panqueue.config.ts
├── enqueue.ts
└── worker.ts
```

Larger projects can put the config in a shared internal package:

```txt
.
├── packages
│   └── queue-config
│       └── index.ts
├── apps
│   ├── api
│   └── worker
└── deno.json
```

Export only the config and the queue type from that package:

```ts title="packages/queue-config/index.ts"
import { definePanqueueConfig } from "@panqueue/config";

export type Queues = {
  email: { to: string; subject: string; body: string };
  thumbnail: { imageUrl: string; width: number };
};

export const pq = definePanqueueConfig<Queues>({
  redis: { url: "redis://localhost:6379" },
  queues: {
    email: {},
    thumbnail: {},
  },
});
```

<Callout type="warn" title="Keep config import-safe">
  The config file should define values, not start workers or enqueue jobs. That keeps imports safe
  in producer processes, worker processes, tests, and scripts.
</Callout>

## API reference

```ts
function definePanqueueConfig<TQueues extends QueueMap>(
  config: PanqueueConfig<TQueues>,
): PanqueueConfig<TQueues>;
```

| Export                    | Kind     | Description                                                         |
| ------------------------- | -------- | ------------------------------------------------------------------- |
| `definePanqueueConfig`    | Function | Defines a typed Panqueue config and returns it unchanged.           |
| `PanqueueConfig<TQueues>` | Type     | Shared config object with `redis` and `queues`.                     |
| `QueueConfig`             | Type     | Per-queue settings inside `queues`.                                 |
| `RetentionRule`           | Type     | Per-state retention policy: `false`, `true`, or `{ ttl?, count? }`. |

```ts
interface PanqueueConfig<TQueues extends QueueMap> {
  redis: ConnectionOptions;
  queues: { [K in keyof TQueues]: QueueConfig };
}

interface QueueConfig {
  retention?: {
    completed?: RetentionRule;
    failed?: RetentionRule;
  };
}
```
